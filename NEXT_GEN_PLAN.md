# NEXT GEN PLAN — битви AAA-рівня, населені планети, 3D-життя

> Контекст рішення: веб-версія для гравців скасована. Єдина платформа — мобільна
> (Android + iOS через Capacitor). Це знімає обмеження "має працювати у браузері"
> і робить доступними нативні опції. Tier `ultra` (desktop web) стає мертвим кодом.

---

## A. Космічні битви: AAA-відчуття при МЕНШІЙ нагрузці

### Діагноз поточного стану

Зараз існують ДВА паралельні рушії рейду:

| Рушій | Файл | Що робить | Проблема |
|---|---|---|---|
| Three.js RaidEngine | `game/raid/RaidEngine.ts` (1397 рядків) | GLB-кораблі, PNG-фон, повна 3D-сцена | Важкий: по draw call на кожен дрон/снаряд, без інстансингу, GLTF-парсинг на старті |
| Pixi fallback | `game/raid/PixiRaidEngine.ts` (678 рядків) | Плоскі процедурні полігони | Дешевий, але виглядає бідно: нема вибухів, трейлів, частинок, shake |
| SpaceArena | Three.js, окремий код | Дуельний режим | Дублювання логіки AI/фізики з рейдом |

Головна причина "печально І важко" одночасно: зусилля розмазані на два рушії,
жоден не доведений. Three-версія платить за справжнє 3D, яке на маленькому
екрані телефона майже не відрізняється від якісного 2.5D.

### Рекомендація: один Pixi v8 рушій + "дешевий преміум"

НЕ Unity. Обґрунтування нижче. Стратегія — консолідувати все бойове на PixiJS v8
і вкласти зекономлене в якість ефектів, бо AAA-відчуття на телефоні на 80%
складається з дешевих трюків, а не з геометрії:

1. **Спрайт-бейкінг замість живого 3D.** Наявні GLB кораблів (включно з
   кастомними Tripo-кораблями гравців) рендеряться ОДИН раз при вході в бій
   у sprite-атлас: 16 кутів повороту × 2-3 стани (норма/пошкоджений/буст)
   через offscreen Babylon/Three в RenderTexture. Далі бій оперує спрайтами.
   Виглядає як 3D, коштує як 2D. Кастомні кораблі гравців зберігаються.
2. **Бюджет частинок через ParticleContainer** (Pixi v8 інстансинг):
   двигунні трейли, іскри попадань, уламки. Ліміти по тирах:
   low 150 / mid 400 / high 900 частинок. Пул, нуль алокацій per-frame.
3. **Вибухи — процедурно згенеровані sprite-sheets.** Генеруються один раз
   при білді (скрипт → webp-атлас у bundle), у грі — програвання кадрів.
   Це найбільший візуальний стрибок за найменші гроші.
4. **Преміум-фідбек, який коштує ~0:** screen shake (позиція камери),
   hit-stop 2-3 кадри при вбивстві, білий flash-tint при попаданні,
   slow-mo 0.3с на останньому ворогу хвилі, глибокий SFX-шар.
5. **Фон:** один статичний webp + 2 шари зірок, що рухаються тільки в межах
   бойової сцени (у бою своя камера, конфлікту з CameraController галактики
   немає — але звіритись з GAME_BIBLE перед реалізацією).
6. **Динамічна роздільність:** якщо frame time > 20мс три кадри поспіль —
   знизити resolution до 0.8×, повернути при стабілізації. Непомітно оку,
   рятує слабкі пристрої.
7. **Прибрати:** Graphics.clear()+redraw на кожен постріл снаряда
   (PixiRaidEngine.fireProjectile) — замінити на 2 спільні текстури.
   Викинути tier 'ultra' та десктопні гілки після консолідації.

### Чому не Unity (as a Library)

Тепер, без веба, UaaL технічно чистий варіант: Unity-модуль в Capacitor-додаток,
відкривається як окрема активність. Але для соло-розробника:

- +30-60 МБ до AAB/IPA (зараз гра важить помірно — це подвоєння);
- другий toolchain: Unity-проєкт, ліцензія, білд-пайплайн поверх
  Android Studio/Xcode, синхронізація версій;
- міст даних (нагороди, стан гравця) — окремий шар багів;
- арт-стиль гри процедурно-мінімалістичний — Unity-графіка "з коробки"
  випадатиме зі стилю, а доводити її до стилю = ті самі тижні роботи.

Спрайт-бейкінг + ефекти дають ~80% візуального стрибка за ~10% ціни.
Повернутись до Unity варто лише якщо повноцінний 3D-dogfight стане
центральною механікою гри, а не одним із режимів.

### Етапи і оцінка (соло, тижні)

1. Тиждень 1: інфраструктура — GLB→sprite-атлас бейкер, пул частинок,
   вибухові sprite-sheets, динамічна роздільність.
2. Тиждень 2: переписати рейд на єдиний рушій, викинути Three-гілку.
3. Тиждень 3: ефект-пас (shake/hit-stop/трейли/SFX), тюнінг тирів.
4. Тиждень 4 (опційно): арена на той самий рушій, видалення дубльованого AI.

Ризик: якість бейкнутих спрайтів кастомних Tripo-кораблів (текстури standard).
Мітигація: рендер з фіксованим стильним освітленням + rim light — вирівнює.

## B. Населені планети: захоплення → колонізація

### Тон (важливо для науково-освітньої ідентичності)

Пряме "завоювання аборигенів" конфліктує з тоном гри (освіта, наука, дослідження).
Рішення — два класи життя:

- **Біосфери** (нерозумна фауна/флора) — колонізація як зараз, без змін.
- **Цивілізації** (рідкісні, ~3-5% населених планет у Core Zone) — не
  "захоплення", а **"інтеграція в Зоряну Спільноту"** трьома шляхами:
  1. **Дипломатія** — обмін знаннями, серія місій/квізів (в дусі Академії),
     повільно, без бою, найбільший бонус довіри;
  2. **Демонстрація сили** — орбітальна облога без жертв: блокада + злам
     оборони через бойовий рушій з розділу A, швидше, стартовий штраф довіри;
  3. **Інфільтрація** — через tech-гілку bio/chem, середній шлях.

### Механіка

- **Генерація:** детерміновано з seed планети (SeededRNG, правило "один seed —
  один результат"). Цивілізація = { ера технологій (кам'яна → індустріальна →
  орбітальна), населення, темперамент (мирний/насторожений/войовничий),
  естетика для генерації зображень }. Темперамент визначає доступність шляхів.
- **Виявлення:** через існуючі planet missions — orbital_scan показує аномалії
  (світло міст, радіосигнали), orbital_probe підтверджує цивілізацію.
  Вписується в reveal levels без нової системи.
- **Після інтеграції:** населення = робоча сила колонії (множник до виробництва
  будівель) + нова шкала **"Довіра"** (0-100): росте від виконання місій
  спільноти, падає від агресивних дій; при низькій — страйки (виробництво ×0.5),
  події невдоволення.
- **Прогресія:** відкривається з L20+ (Фаза 3, Core Zone) + tech node
  "Ксенодипломатія" в bio-гілці. Населені планети генеруються тільки в
  core-системах (depth ≥ 2) — не ламає баланс перших кілець.

### Дані

- `packages/core/src/types/civilization.ts`: `Civilization`, `IntegrationPath`,
  `TrustState`.
- Стан інтеграції — в `game_state` JSONB (без нових таблиць у MVP).
- Серверна валідація результату облоги (антічіт) — по патерну існуючих api.

### Етапи і оцінка

1. Тиждень 1-2: генерація цивілізацій + виявлення через місії + UI картки
   цивілізації (MVP: подивитись і подружитись через дипломатію).
2. Тиждень 3-4: шкала довіри, робоча сила, події.
3. Тиждень 5-6: облога через новий бойовий рушій + інфільтрація.

Залежність: розділ 3 краще робити ПІСЛЯ розділу A (бойовий рушій).

## C. 3D-життя на планетах: картинка → Tripo → живі істоти

### Що вже є (перевикористовуємо ~70%)

- Kling/Gemini image-генерація з квark-гейтом і модерацією — є.
- Tripo `image_to_model` + polling → GLB (face_limit 5000, smart_low_poly) —
  є для кораблів (`tripo-client.ts`, `ship_models` в Neon).
- Babylon.js GLB-в'ювер (`SurfaceBabylonView`, `BuildingGLBLoader`) — є.

### Чого бракує

1. **Анімація.** Tripo вміє rig + retarget з пресетними анімаціями
   (idle/walk/run) — окремі task-типи (`animate_prerigcheck` → `animate_rig` →
   `animate_retarget`). Треба додати в tripo-client. Не кожна згенерована
   істота рігається успішно — потрібен fallback: істота без анімації отримує
   процедурний рух (погойдування/поворот/переміщення ковзанням).
2. **Власне сховище GLB.** Tripo-URL тимчасові. Для кораблів це вже ризик,
   для істот критично: качати GLB на свій сторедж (Vercel Blob) одразу після
   success і зберігати свій URL. Виправити і для кораблів заодно.
3. **Сцена "життя".** Новий екран "Біосфера" на Babylon: процедурний
   терен-патч у палітрі планети (тип планети → біом), 1-3 істоти з
   idle/walk-циклами і простим wander-AI (крапка призначення → йти → стояти).
   Вхід з planet view / колонії. НЕ вбудовувати в HexSurface у MVP —
   окрема сцена, як ship viewer.

### Вартість і монетизація

- Tripo: модель + rig + retarget ≈ $0.25-0.45 за істоту.
- Пропозиція: **1 безкоштовна істота на акаунт** (onboarding wow-момент,
  двигун сарафану — скріншоти/відео своєї істоти), далі 50-70⚛ за істоту
  або tier-1 реклама. Ліміт: 3 істоти на планету (і бюджет рендера, і сенс).
- Генерація зображення істоти — через існуючий life-forms пайплайн
  (Вектор B з TODO синергія: це фактично його 3D-розширення).

### Бюджет продуктивності (mid-phone)

- ≤ 3 істоти × 5000 полігонів + терен ≤ 8k полігонів;
- 1 directional light, без тіней на low/mid (blob-shadow текстурою);
- 30fps cap у сцені біосфери (це споглядальний екран, не бій);
- вивантаження сцени при виході (Babylon dispose повний).

### Етапи і оцінка

1. Тиждень 1: Vercel Blob сховище GLB (+кораблі), `creature_models` таблиця,
   image→model ланцюг для істот (переюз ship-пайплайну).
2. Тиждень 2: сцена "Біосфера" — терен, одна статична істота, орбітальна
   камера. Це вже релізний MVP.
3. Тиждень 3-4: rig+retarget анімації, wander-AI, до 3 істот, SFX/ambient.
4. Пізніше: істоти як міні-спрайти (impostors) на HexSurface біля будівель,
   зв'язка з розділом B (естетика цивілізації → генерація її представників).

Ризики: нестабільність Tripo-рігінгу (мітигація — процедурний fallback);
вартість при вірусному попиті (мітигація — жорсткий квark-гейт + ліміти).

## Порядок виконання (загальна рекомендація)

1. **A (битви)** — перший: це фундамент і для облог з B, і найпомітніший
   стрибок якості для поточних гравців. ~3-4 тижні.
2. **C-MVP (біосфера, тижні 1-2)** — паралельно з хвостом A можливо, бо
   не перетинаються кодом. Дає wow-контент для маркетингу до софт-лончу.
3. **B (цивілізації)** — після A, з облогою на новому рушії. ~4-6 тижнів.
4. **C-повний (анімації, wander)** — після B-MVP.

Разом: ~3 місяці соло-розробки до повного набору, з релізними точками
кожні 2-3 тижні (консолідований бій → біосфера MVP → дипломатія → облога).
